# 文件链路深度讲解

本文讲清楚这套仓库里的文件上传、预签名、下载、删除，为什么这么设计。

## 一、先看结论

文件模块不是单纯“上传文件”，而是三件事同时做：

1. 把文件交给具体存储器保存。
2. 把文件元数据存进数据库。
3. 给前端提供可直传、可下载、可删除的统一接口。

核心代码在：

- `yudao-module-infra/src/main/java/cn/iocoder/yudao/module/infra/controller/admin/file/FileController.java`
- `yudao-module-infra/src/main/java/cn/iocoder/yudao/module/infra/service/file/FileServiceImpl.java`

## 二、入口层：FileController

`FileController` 的职责是把 HTTP 行为拆开。

它支持几种模式：

- `/upload`：后端直接接收文件再上传
- `/presigned-url`：前端拿预签名地址直传对象存储
- `/create`：前端直传后回调记录元数据
- `/get`、`/delete`、`/page`：查询、删除、分页
- `/{configId}/get/**`：按配置和路径下载文件

你要注意：**Controller 不知道具体存到哪里，它只负责把请求转成 service 调用。**

## 三、业务层：FileServiceImpl

### 1. 先算文件路径

`generateUploadPath(name, directory)` 会把目录、日期、后缀拼起来。

它这么做的目的很简单：

- 避免重名覆盖
- 方便按日期归档
- 便于存储器按目录管理

### 2. 再上传到具体存储器

`createFile(byte[] content, ...)` 会先拿 `FileClient`，再调用 `client.upload(...)`。

这说明仓库把“存储协议”抽象成了 `FileClient`，而不是把 OSS、S3、FTP、本地磁盘写死在业务里。

### 3. 最后落库

上传成功后，`FileDO` 会写入数据库。

数据库里保存的不是文件内容，而是：

- 名称
- 路径
- URL
- 类型
- 大小
- 配置编号

这就是典型的“内容在对象存储，索引在数据库”。

## 四、下载为什么要 `@TenantIgnore`

`getFileContent(...)` 这个下载接口标了 `@TenantIgnore`。

原因是下载路径本身往往是公开资源或者对象存储资源，不应该强行套租户条件。

这正好说明：

- 租户不是万能过滤器
- 有些资源天然不适合走租户限制

## 五、删除为什么要先删存储再删记录

`deleteFile(...)` 和 `deleteFileList(...)` 都先调用 `client.delete(path)`，再删数据库记录。

原因是：

- 数据库记录只是索引
- 真正占空间的是文件内容
- 如果只删数据库，文件会泄漏在存储里

## 六、你该掌握的 Java / 架构知识

### 1. 适配器思想

`FileClient` 就是适配器抽象，不同存储实现不同接口。

### 2. 元数据与内容分离

这是企业系统里非常常见的设计。

### 3. 幂等与唯一性

路径生成必须保证唯一，否则会覆盖。

### 4. 接口分层

上传、预签名、记录、下载、删除不能混在一个方法里，否则后续很难维护。

## 七、建议你的阅读顺序

1. `FileController`
2. `FileServiceImpl`
3. `FileClient` 相关实现
4. `FileDO` 和 `FileMapper`

看完你就知道文件能力为什么可以同时支持后端上传和前端直传。
