# 从零新增一个业务模块实战教程

这份文档不只是告诉你“建几个类”，而是教你在这套仓库里，如何像一个高级工程师一样新增一个完整业务模块。

为了讲透，我用一个具体例子：`yudao-module-project`。

假设你现在要做一个“项目管理”模块，里面第一期先落一个最基础的功能：项目立项单。

## 一、先做架构判断，不要先写代码

开始前先回答 4 个问题：

1. 这个功能能不能放进现有 `system / infra / bpm`？
2. 如果不能，它是不是独立业务域？
3. 它需不需要审批流？
4. 它需不需要文件、租户、数据权限、字典、菜单、角色这些底座能力？

如果答案是：

- 这是独立业务域
- 后续会持续扩展
- 需要自己的表、自己的页面、自己的服务

那就应该新建模块，而不是硬塞到 `system` 里。

## 二、模块应该长什么样

你应该对齐现有模块风格，而不是自己发明目录。

一个标准模块一般会有：

- `controller/admin`
- `service`
- `dal/dataobject`
- `dal/mysql`
- `convert`
- `enums`
- 必要时 `api`

也就是说，**先学现有骨架，再扩新功能**。

## 三、第一步：把模块接进 Maven

需要做的事：

1. 在根 `pom.xml` 加入 `yudao-module-project`
2. 新建模块自己的 `pom.xml`
3. 在 `yudao-server/pom.xml` 引入新模块依赖

为什么这么做：

- 根 `pom` 决定聚合构建
- 模块 `pom` 决定自己的依赖边界
- `yudao-server` 决定最终有没有把它装进可运行应用

## 四、第二步：先设计表，不要先写 Controller

以“项目立项单”为例，你至少要想清楚：

- 主键
- 项目编号
- 项目名称
- 项目类型
- 项目经理
- 立项日期
- 状态
- 备注
- 租户字段是否需要
- 审计字段是否复用 `BaseDO`

如果未来还要走审批，就要提前考虑：

- 是否需要 `processInstanceId`
- 状态是业务状态还是审批状态，还是两者并存

## 五、第三步：先写 DO 和 Mapper

### 1. DO

`ProjectInitiationDO` 负责映射数据库表。

它应该：

- 继承 `BaseDO`，复用创建时间、更新时间、逻辑删除等字段
- 只表达表结构，不写业务逻辑

### 2. Mapper

`ProjectInitiationMapper` 负责持久化。

原则是：

- 能复用 `BaseMapperX` 就复用
- 简单分页、查询按现有模式来
- 复杂联表再单独补 SQL

## 六、第四步：写 VO，不要把 DO 直接暴露给前端

至少会有三类对象：

- `ProjectInitiationCreateReqVO`
- `ProjectInitiationUpdateReqVO`
- `ProjectInitiationRespVO`

为什么必须拆：

- 前端输入字段和数据库字段不完全一样
- 更新、创建、详情的约束不一样
- 不能让前端直接感知数据库内部结构

## 七、第五步：写 Service，真正的业务编排放这里

比如 `ProjectInitiationServiceImpl`：

- `createProjectInitiation()`
- `updateProjectInitiation()`
- `getProjectInitiation()`
- `getProjectInitiationPage()`

Service 该做：

- 参数校验
- 状态校验
- 事务控制
- 调 Mapper
- 调其他模块 API
- 触发事件

Service 不该做：

- 拼 HTTP 响应
- 直接依赖前端传来的展示字段

## 八、第六步：写 Controller，只做接口装配

比如：

- `POST /project/initiation/create`
- `PUT /project/initiation/update`
- `GET /project/initiation/get`
- `GET /project/initiation/page`

Controller 要配：

- `@RequestMapping`
- `@Operation`
- `@Validated`
- `@PreAuthorize`

这一步要同步想好 permission 标识，例如：

- `project:initiation:create`
- `project:initiation:update`
- `project:initiation:query`

## 九、第七步：如果要接 BPM，怎么接

如果“项目立项单”要审批，就参考 OA 请假样板：

1. 先保存业务单。
2. 拿业务单 id 作为 `businessKey`。
3. 调 `BpmProcessInstanceApi.createProcessInstance(...)` 发起流程。
4. 把 `processInstanceId` 回写业务表。
5. 审批结束后监听事件，更新业务状态。

这里最重要的认知是：

- 业务数据归你自己的业务表
- 流程流转归 BPM 模块
- 两者通过 `businessKey` 绑定

## 十、第八步：把菜单和权限补齐

后端接口写完还不够。

你还要让系统知道这组接口对应什么资源。

通常要补：

- 菜单
- 按钮权限点
- 角色授权入口

否则接口存在，前端也可能根本配不到。

## 十一、第九步：前端不是附属品，而是同一条交付链

后端新功能最终要给前端使用，所以还要补：

- API 封装
- 列表页
- 创建页
- 编辑页
- 详情页
- 权限按钮控制

如果是字典字段，还要对齐字典数据。

## 十二、第十步：测试和验证怎么做

至少做三类验证：

1. 启动能不能过
2. 接口能不能调通
3. 如果带审批，流程能不能发起并回写状态

如果是多租户场景，还要补：

- tenantId 是否正确
- 权限是否正确
- 数据权限是否误伤

## 十三、最容易踩的坑

### 1. 把模块硬塞进 `system`

结果是系统底座和业务域越来越混。

### 2. 直接暴露 DO

后续一改表结构，接口就容易崩。

### 3. 忘了补菜单权限

接口写了，页面却访问不了。

### 4. 忘了事务

业务表落了，流程没发成功，数据就裂了。

### 5. 忽略租户和数据权限

本地单租户能跑，上线多租户就炸。

## 十四、你以后应该形成的模块化思维

新增模块不是“多建几个类”，而是一次完整的架构动作：

- 定义业务边界
- 接入系统底座
- 保持模块内聚
- 控制跨模块依赖
- 与权限、租户、流程、文件等横切能力正确协作

如果你能按这份教程把一个新模块完整落一次，这套仓库你就不是“会改功能”，而是开始具备真正的模块化架构能力了。
