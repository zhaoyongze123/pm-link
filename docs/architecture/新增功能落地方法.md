# 新增功能落地方法

这份文档告诉你：如果要在这套仓库里新增一个功能，应该怎么从需求一路落到代码。

## 一、先判断功能归属

先问自己四个问题：

1. 这是系统管理能力吗？如果是，优先看 `system`。
2. 这是基础设施能力吗？如果是，优先看 `infra`。
3. 这是审批/流程能力吗？如果是，优先看 `bpm`。
4. 这是全新业务域吗？如果是，考虑新建 `yudao-module-xxx`。

## 二、再判断依赖能力

一个新功能通常会依赖这些底座：

- 登录认证：`security`
- 租户隔离：`tenant`
- 数据权限：`biz-data-permission`
- 数据库访问：`mybatis`
- 事件解耦：Spring Event
- 文件：`infra/file`
- 审批：`bpm api`

## 三、标准分层怎么写

### 1. Controller

只做：

- 接收请求
- 参数校验
- 权限控制
- 返回响应

### 2. Service

只做：

- 业务编排
- 状态校验
- 事务控制
- 调用其他模块 API
- 触发事件

### 3. DO / Mapper

只做：

- 表结构映射
- 数据持久化
- 条件查询

### 4. VO / DTO / Convert

- `VO` 给前端
- `DTO` 在模块间传递
- `Convert` 负责对象转换

## 四、如果是 BPM 类功能，怎么落

以“项目合同审批”为例：

1. 先建业务表。
2. 再建 `DO`、`Mapper`。
3. 再建 `Controller`、`Service`、`VO`。
4. 业务保存后调用 BPM API 发起流程。
5. 用 `businessKey` 绑定单据和流程。
6. 审批结束后通过事件监听回写状态。

## 五、如果是文件类功能，怎么落

1. 先确认是后端上传还是前端直传。
2. 如果需要对象存储，优先复用 `FileClient`。
3. 文件内容与元数据分离。
4. 下载接口注意租户和权限边界。

## 六、如果是权限类功能，怎么落

1. 先判断是认证、租户还是数据权限。
2. 再决定是改过滤器、AOP、还是 SQL 拦截器。
3. 不要把权限逻辑写进业务方法里硬编码。

## 七、最容易犯的错

- 业务数据和流程状态混在一起。
- Controller 写太多业务。
- 直接把 DO 暴露给前端。
- 忽略事务边界。
- 忽略租户和数据权限上下文。
- 新功能乱放模块，边界越来越脏。

## 八、你应该形成的判断习惯

以后每接一个需求，先问：

- 这个功能属于哪个模块？
- 它要依赖哪些底座？
- 哪些状态要落库？
- 哪些状态只存在上下文？
- 哪些逻辑应该事件化？
- 哪些逻辑应该事务化？

如果你能把这几个问题答稳，你就不是在“写功能”，而是在“设计系统”。
