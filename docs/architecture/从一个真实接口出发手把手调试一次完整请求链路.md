# 从一个真实接口出发，手把手调试一次完整请求链路

这份文档不是讲概念，而是带你顺着一个真实接口，把请求从浏览器一路追到数据库。

我用两个真实接口做样板：

- 登录接口：`POST /system/auth/login`
- 文件上传接口：`POST /infra/file/upload`

## 一、先看请求进来之前发生了什么

在请求打进 Controller 之前，系统先经过：

- Web 前缀路由
- 租户过滤
- 安全过滤
- token 解析
- 全局异常处理
- 响应包装记录

所以你调试任何接口，第一步都不是看 Service，而是看请求有没有在 Filter 阶段就被拦掉。

## 二、以登录接口为例

### 1. 前端发请求

前端提交账号、密码、验证码等字段。

### 2. Controller 接收

`AuthController.login()` 接收 `AuthLoginReqVO`。

这里的关键不是“接到了参数”，而是：

- 路径是否对
- 请求是否走到了 `/system/auth/login`
- `@PermitAll` 是否让它跳过登录校验

### 3. Service 处理

`AdminAuthServiceImpl.login()` 会：

- 校验验证码
- 校验账号密码
- 校验用户状态
- 记录登录日志
- 创建 token

### 4. Token 创建

`OAuth2TokenServiceImpl.createAccessToken()` 把 access token 和 refresh token 落到 MySQL 和 Redis。

### 5. 响应返回

Controller 返回 `CommonResult<AuthLoginRespVO>`。

`GlobalResponseBodyHandler` 会把这个返回结果记录到请求上下文，方便后续 API 日志使用。

## 三、以文件上传接口为例

### 1. 前端发请求

上传表单提交 `MultipartFile`。

### 2. Controller 接收

`FileController.uploadFile()` 接收 `FileUploadReqVO`。

这里它先做的不是直接调用存储器，而是：

- 读取文件内容
- 取原文件名
- 取目录参数
- 调 `FileService.createFile(...)`

### 3. Service 处理

`FileServiceImpl.createFile(...)` 会：

- 补 MIME 类型
- 补文件名后缀
- 生成唯一路径
- 调 `FileClient.upload(...)`
- 把文件元数据落库

### 4. 返回结果

返回的是文件 URL，而不是文件对象本身。

## 四、调试链路时该怎么看

### 1. 先看 Filter

如果接口返回 401/403，先查：

- token 有没有带上
- tenantId 有没有带上
- userType 对不对

### 2. 再看 Controller

如果接口 404，查：

- 路径前缀对不对
- 请求方法对不对
- 包路径有没有被自动挂上 `/admin-api` 或 `/app-api`

### 3. 再看 Service

如果业务失败，查：

- 校验逻辑
- 事务边界
- 外部 API 调用
- 数据库写入

### 4. 最后看日志

如果还是不明白，看：

- 操作日志
- API 访问日志
- 异常日志
- traceId

## 五、你要掌握的排查思路

### 1. 请求有没有到后端

先排除前端没发出去、网关没转发、路由不对。

### 2. 请求有没有过过滤器

token、租户、安全、响应包装都在这里。

### 3. 请求有没有进 Controller

看路径、方法、注解。

### 4. 请求有没有进 Service

看日志、断点、异常。

### 5. 数据有没有落库

看 Mapper、事务、主键、回滚。

## 六、你应该形成的调试习惯

不要一上来就盯业务方法。

真正正确的顺序是：

1. 网络层
2. Filter 层
3. Controller 层
4. Service 层
5. Mapper 层
6. 数据库
7. 日志与事件

这才是一个完整请求链路的真实排查顺序。
