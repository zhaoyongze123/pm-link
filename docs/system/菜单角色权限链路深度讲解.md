# 菜单、角色、权限链路深度讲解

这份文档讲清楚一个很多人一开始最容易混的系统能力：菜单、角色、权限，到底是怎么串起来的。

## 一、先看结论

这套仓库的权限模型不是“用户直接拥有哪些接口权限”，而是一个标准的链式模型：

1. 用户绑定角色。
2. 角色绑定菜单。
3. 菜单上声明 permission 标识。
4. Controller 上通过 `@PreAuthorize("@ss.hasPermission('xxx')")` 校验。
5. 安全服务把当前用户的角色和菜单映射关系算出来，判断是否放行。

所以核心关系不是“用户 -> 权限”，而是：

`用户 -> 角色 -> 菜单 -> permission -> 接口`

## 二、入口层：Controller 怎么声明权限

你看 `RoleController` 和 `MenuController`：

- `RoleController.createRole()` 用 `@PreAuthorize("@ss.hasPermission('system:role:create')")`
- `MenuController.createMenu()` 用 `@PreAuthorize("@ss.hasPermission('system:menu:create')")`

这说明接口权限不是写在配置文件里，而是**直接跟着接口走**。

这样做的好处是：

- 看接口代码时就知道访问条件
- 权限点和业务接口不会脱节
- 前后端的 permission 标识能统一

## 三、菜单为什么不仅是前端菜单

很多初学者以为菜单只是“左侧导航”。这套仓库不是。

在 `MenuDO` 的注释里已经说明，菜单还承担一个职责：

- 给后端接口提供 permission 标识

也就是说，菜单同时承载：

- 前端展示结构
- 路由能力
- 按钮权限
- 后端接口权限映射

所以这里的“菜单”本质上是 **权限资源模型**，不是纯 UI 概念。

## 四、角色做什么

`RoleController` 负责角色的增删改查。

但角色本身不直接知道用户有哪些接口权限，它只知道：

- 自己有哪些菜单
- 自己的数据权限范围是什么

因此角色在体系里扮演的是“权限集合”的角色。

## 五、PermissionServiceImpl 才是权限计算核心

真正的核心代码在：

`yudao-module-system/src/main/java/cn/iocoder/yudao/module/system/service/permission/PermissionServiceImpl.java`

### 1. `hasAnyPermissions(...)` 怎么算

它的大致逻辑是：

1. 先拿当前用户启用状态下的角色。
2. 再根据 permission 找到对应菜单 id。
3. 再根据菜单 id 找到拥有哪些角色。
4. 判断当前用户角色集合和菜单角色集合是否有交集。

这是一个典型的“反查式权限判定”。

不是“用户身上直接挂一串 permission 字符串”，而是通过角色和菜单关系动态推导。

### 2. 为什么要先查菜单 id

因为 permission 标识最终是挂在菜单上的。

这样有两个好处：

- 前端菜单和后端权限点能统一维护
- 一个 permission 可以和菜单体系保持一致，不会散落在各处

### 3. 超管为什么特殊

`roleService.hasAnySuperAdmin(...)` 会让超管直接通过。

这说明权限体系里有“显式超管兜底”的设计，而不是所有权限都穷举配置。

## 六、用户角色、角色菜单是怎么分配的

### 1. 给用户分角色

`assignUserRole(userId, roleIds)` 做的是用户和角色中间表的增删。

它不是覆盖式暴力重建，而是先比对数据库已有值，再分别新增和删除差异项。

这体现了两个工程点：

- 避免无意义写库
- 保持关系表更新清晰可控

### 2. 给角色分菜单

`assignRoleMenu(roleId, menuIds)` 也是同样思路。

并且它会清理相关缓存。

说明权限系统真正复杂的地方不只是“关系表写入”，而是**缓存一致性**。

## 七、`@ss.hasPermission(...)` 最后怎么生效

`@ss` 背后的实现会走 `SecurityFrameworkServiceImpl`。

核心逻辑是：

1. 从安全上下文拿当前登录用户 id。
2. 通过 `PermissionCommonApi` 去系统模块查权限。
3. 返回是否拥有指定 permission。

也就是说，Spring Security 负责拦截，真正的权限判定还是在你自己的系统权限服务里完成。

## 八、数据权限和菜单权限不要混

很多人会混：

- `@PreAuthorize` 校验的是“能不能访问接口”
- `@DataPermission` 控制的是“进了接口以后能看到哪些数据”

它们不是一个层次。

- 菜单/角色/permission：接口访问权限
- 数据权限：数据可见范围

## 九、你该掌握的 Java / 架构知识

### 1. RBAC

这是标准的基于角色的访问控制模型。

### 2. 中间表建模

用户角色、角色菜单本质上都是多对多关系。

### 3. 缓存一致性

权限数据变化后要清缓存，否则判断会错。

### 4. Spring Security 表达式

`@PreAuthorize` 是声明式权限控制，不是手写 if/else。

## 十、你以后读权限链路的顺序

建议这样读：

1. `MenuController`
2. `RoleController`
3. `PermissionServiceImpl`
4. `SecurityFrameworkServiceImpl`
5. `AuthController.getPermissionInfo()`

把这条链路看懂，你就会真正明白：为什么一个按钮能不能点、一个接口能不能调，不是拍脑袋判断，而是整套角色菜单模型算出来的。
