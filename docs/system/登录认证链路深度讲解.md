# 登录认证链路深度讲解

本文只讲一条链路：用户从浏览器登录，到后端签发 token，再到后续请求如何恢复成当前用户。

## 一、先看结论

这套仓库的登录不是 Session，也不是把用户状态长期挂在服务端内存里，而是 **OAuth2 token + Redis + MySQL** 的无状态方案。

核心链路是：

1. 前端提交账号密码。
2. `AuthController` 接收请求。
3. `AdminAuthServiceImpl` 校验验证码、账号、密码、状态。
4. `OAuth2TokenServiceImpl` 创建 access token 和 refresh token。
5. token 写入数据库，并同步进 Redis。
6. 后续请求进入 `TokenAuthenticationFilter`。
7. filter 通过 token 查询用户信息，组装 `LoginUser`，放入安全上下文。
8. Controller 和 Service 通过上下文拿当前登录人。

## 二、入口层：AuthController

对应代码：`yudao-module-system/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/AuthController.java`

这里的职责很纯粹：

- 定义接口路径
- 接收参数
- 标记 `@PermitAll` 或权限注解
- 调用 service
- 返回 `CommonResult`

你要记住一个原则：**Controller 不处理核心业务，只做装配和转发。**

以 `/login` 为例，控制器只把 `AuthLoginReqVO` 交给 `authService.login(reqVO)`。

## 三、业务层：AdminAuthServiceImpl

对应代码：`yudao-module-system/src/main/java/cn/iocoder/yudao/module/system/service/auth/AdminAuthServiceImpl.java`

这个类才是真正的登录编排层。

### 1. 登录前先校验验证码

`validateCaptcha(reqVO)` 先检查图形验证码是否通过。

这一步属于安全前置校验，避免暴力撞库。

### 2. 再校验账号、密码、状态

`authenticate(username, password)` 做三件事：

- 查用户是否存在
- 校验密码是否匹配
- 校验用户是否禁用

这一步的关键不是语法，而是业务顺序：

- 账号不存在，直接失败
- 密码错，直接失败
- 用户禁用，直接失败

每种失败都会写登录日志。

### 3. 登录成功后创建 token

`createTokenAfterLoginSuccess(...)` 会：

- 写登录成功日志
- 调用 `oauth2TokenService.createAccessToken(...)`
- 把返回的 token 对象转换成登录响应

所以真正签发 token 的责任不在 `AuthController`，也不在 `TokenAuthenticationFilter`，而在 `OAuth2TokenServiceImpl`。

## 四、令牌层：OAuth2TokenServiceImpl

对应代码：`yudao-module-system/src/main/java/cn/iocoder/yudao/module/system/service/oauth2/OAuth2TokenServiceImpl.java`

### 1. 为什么要同时有 access token 和 refresh token

- `access token` 用来访问接口，生命周期短
- `refresh token` 用来续期，生命周期长

这是典型的前后端分离登录设计。

### 2. 创建令牌时做了什么

`createAccessToken(...)` 里先验证客户端，再创建 refresh token，再创建 access token。

它写入两层存储：

- MySQL：持久化，便于查询和审计
- Redis：加速 token 校验

### 3. 为什么要写用户信息

`buildUserInfo(...)` 会把昵称、部门等信息塞进 token 关联数据。

这样后续请求恢复 `LoginUser` 时，不必每次都回表查用户基础信息。

### 4. 为什么要保存 tenantId

创建 access token 时会优先带上 tenantId，避免线程上下文污染导致租户信息丢失。

这说明 token 不只是“登录标识”，还是“上下文载体”。

## 五、请求进入时：TokenAuthenticationFilter

对应代码：`yudao-framework/yudao-spring-boot-starter-security/src/main/java/cn/iocoder/yudao/framework/security/core/filter/TokenAuthenticationFilter.java`

### 1. 过滤器做什么

它只做一件事：**把请求里的 token 变成当前用户上下文。**

### 2. 主要步骤

- 从请求头或参数中取 token
- 调 `oauth2TokenApi.checkAccessToken(token)` 校验 token
- 校验成功后构建 `LoginUser`
- 调 `SecurityFrameworkUtils.setLoginUser(...)`
- 如果异常，统一交给全局异常处理器输出 JSON

### 3. 为什么 filter 不直接抛异常

因为有些接口本来就是匿名可访问的。

所以 token 无效时，它会返回 `null`，而不是直接把整个请求打死。

## 六、你真正要掌握的 Java 知识点

### 1. 分层思想

Controller、Service、DAO 不是形式主义，是为了把“接参”“编排”“持久化”拆开。

### 2. 事务

登录成功后的日志、token 写入、状态更新要在可控边界内完成，避免半成功。

### 3. 依赖注入

Spring 让 filter、service、dao 这些对象由容器装配，代码只关心依赖关系。

### 4. 上下文对象

`LoginUser` 是请求级上下文，不是业务实体。

### 5. 无状态认证

服务端不保存 session，靠 token 恢复身份。

## 七、你以后读登录链路的顺序

建议固定顺序：

1. `AuthController`
2. `AdminAuthServiceImpl`
3. `OAuth2TokenServiceImpl`
4. `TokenAuthenticationFilter`
5. `SecurityFrameworkUtils`

把这条链路看懂，你就真正理解了这套仓库的“人是怎么进系统的”。
