# 通知、邮件、短信链路深度讲解

这份文档把三种常见通知能力串起来讲：站内通知、邮件、短信。

## 一、先看结论

这套仓库里的通知不是单点能力，而是一整套“模板 + 渠道 + 发送记录 + 异步发送”的体系。

核心思想是：

- **通知模板** 定义内容结构。
- **发送服务** 负责组装和调度。
- **消息队列** 负责异步执行。
- **日志表** 负责记录发送结果。

## 二、站内通知链路

系统模块里有：

- `NotifyTemplateServiceImpl`
- `NotifySendServiceImpl`
- `NotifyMessageController`
- 相关 DO、Mapper、API

### 1. 模板先行

通知不是手写字符串，而是先定义模板。

模板会定义：

- 编码
- 名称
- 内容
- 变量
- 适用场景

### 2. 发送服务做什么

`NotifySendServiceImpl` 负责把模板、变量、接收人组装起来，再真正触发发送。

这说明“发通知”不是 Controller 直接调用第三方，而是业务服务统一编排。

## 三、邮件链路

### 1. 核心入口

邮件相关的核心类包括：

- `MailSendApi`
- `MailSendApiImpl`
- `MailSendServiceImpl`
- `MailTemplateServiceImpl`
- `MailAccountServiceImpl`
- `MailSendConsumer`
- `MailProducer`

### 2. 发送流程

大致流程是：

1. 调用 `MailSendApi` 发起发送。
2. `MailSendApiImpl` 转到 `MailSendService`。
3. `MailSendServiceImpl` 组装模板、收件人、抄送、密送。
4. 通过 `MailProducer` 发消息。
5. `MailSendConsumer` 消费消息。
6. `MailSendService` 真正执行发送并记录日志。

### 3. 为什么要异步

邮件发送通常不应该阻塞主业务流程。

所以它天然适合：

- 主流程先落库
- 邮件通过 MQ 异步发送
- 发送失败再记录状态和重试

### 4. 为什么要模板化

模板化可以做到：

- 内容复用
- 参数替换
- 统一管理
- 统一审计

## 四、短信链路

### 1. 核心入口

短信相关核心类包括：

- `SmsCodeApi`
- `SmsSendApi`
- `SmsProducer`
- `SmsSendConsumer`
- `SmsCodeService` 相关实现

### 2. 两类短信要区分

短信其实分两种：

- **验证码短信**：登录、注册、重置密码。
- **通知短信**：业务通知、提醒、订阅消息。

这两者虽然都叫短信，但业务语义不同。

### 3. 验证码短信的典型链路

在登录链路里，`AdminAuthServiceImpl` 会通过 `smsCodeApi.sendSmsCode(...)` 和 `smsCodeApi.useSmsCode(...)` 完成发送和校验。

也就是说短信能力不是孤立的，它直接嵌入认证流程。

## 五、你要掌握的 Java / 架构知识

### 1. 模板方法思想

模板先定义变量位，再由发送服务填充。

### 2. 消息队列解耦

真正的发送动作适合异步。

### 3. 发送记录与重试

通知系统最重要的不只是“发出去”，而是“可追踪、可重试、可审计”。

### 4. 服务拆分

模板、账户、发送、消费者要分开，不要写成一个大类。

## 六、你以后读通知链路的顺序

建议顺序：

1. `NotifyTemplateServiceImpl`
2. `NotifySendServiceImpl`
3. `MailSendApiImpl`
4. `MailSendServiceImpl`
5. `MailProducer` / `MailSendConsumer`
6. `SmsCodeApi` / `SmsSendApi`
7. `SmsProducer` / `SmsSendConsumer`

这样你能把通知体系看成一个完整的“模板驱动消息系统”。
