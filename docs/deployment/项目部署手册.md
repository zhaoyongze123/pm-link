# 项目部署手册

本文档面向当前仓库的真实部署体系，目标是让接手人能够基于现有脚本完成 `dev / stage / prod` 三套环境的部署、升级、回滚与排障。

## 1. 文档范围

本文覆盖以下内容：

- 本仓库的部署架构与环境分层
- 本地开发环境启动方式
- `stage` / `prod` 的手工部署流程
- 数据库初始化与增量 SQL 升级流程
- GitHub Actions 自动部署链路
- 常见故障定位方式
- 回滚与变更控制建议

本文不包含任何真实密码、真实密钥、真实数据库连接串或真实业务账号。

## 2. 部署架构概览

当前项目采用“本地构建镜像 + 远程加载镜像 + Docker Compose 启动容器”的交付方式。

核心链路如下：

1. 本地执行 Maven 构建后端产物
2. 本地执行 Docker Build 构建后端和前端镜像
3. 本地通过 `docker save` 导出镜像 tar 包
4. 将镜像、Compose 文件、环境变量文件上传到目标服务器
5. 远程通过 `docker compose` 重建 `server` / `admin` 容器
6. 使用 `actuator/health` 做健康检查
7. 自动清理本地构建产物与远端临时文件

仓库内对应入口：

- 部署主脚本：`script/docker/deploy.sh`
- `107` 兼容入口：`script/docker/deploy-to-107.sh`
- 远端热替换脚本：`script/docker/remote-reload-app.sh`
- 容器编排文件：`script/docker/docker-compose.yml`
- 环境变量目录：`script/docker/env/`

## 3. 环境分层设计

### 3.1 环境职责

| 环境 | 用途 | 特点 |
| --- | --- | --- |
| `dev` | 本地开发 / 联调 | 直接在本机启动，便于调试 |
| `stage` | 生产测试环境 | 用于公网或内网测试验证，部署逻辑最先在这里验证 |
| `prod` | 正式生产环境 | 面向正式用户，要求最严格的变更控制 |

### 3.2 环境变量文件

后端与部署变量：

- `script/docker/env/dev.env`
- `script/docker/env/stage.env`
- `script/docker/env/prod.env`
- `script/docker/env/stage.env.example`
- `script/docker/env/prod.env.example`

前端构建变量：

- `script/docker/env/frontend.dev.env`
- `script/docker/env/frontend.stage.env`
- `script/docker/env/frontend.prod.env`

约定：

- `dev.env` 可直接入库
- `stage.env` / `prod.env` 为真实机密文件，不入库
- `stage.env.example` / `prod.env.example` 只保留非敏感模板
- 手工部署和 CI 部署都统一走 `deploy.sh`

### 3.3 关键变量说明

| 变量 | 作用 |
| --- | --- |
| `SPRING_PROFILES_ACTIVE` | Spring Boot 运行环境 |
| `REMOTE_HOST` | 目标服务器地址 |
| `REMOTE_USER` | 目标服务器登录用户 |
| `REMOTE_DEPLOY_DIR` | 远端部署目录 |
| `MYSQL_ROOT_PASSWORD` | MySQL root 密码 |
| `MASTER_DATASOURCE_URL` | 主数据源连接串 |
| `REDIS_HOST` | Redis 主机名 |
| `SERVER_PORT` | 后端映射端口 |
| `ADMIN_PORT` | 前端映射端口 |
| `FRONTEND_ENV_FILE` | 前端构建时使用的环境文件 |
| `KOD_SSO_*` | 可道云登录与回调配置 |
| `WX_*` | 微信公众号 / 小程序配置 |
| `IMPORT_DATABASE` | 是否在部署时覆盖导入数据库 |

## 4. 运行形态

当前 Docker Compose 编排包含四个核心服务：

| 服务 | 容器名 | 作用 |
| --- | --- | --- |
| MySQL | `ruoyi-mysql` | 主业务数据库 |
| Redis | `ruoyi-redis` | 缓存、分布式能力 |
| Server | `ruoyi-server` | Spring Boot 后端 |
| Admin | `ruoyi-admin` | Vue 管理后台静态站点 |

容器编排文件位置：

- `script/docker/docker-compose.yml`

其中有两个关键设计：

1. `mysql` 首次启动时会挂载 `sql/mysql/ruoyi-vue-pro.sql` 做初始化
2. `server` 通过 `ARGS` 注入大部分外部配置，避免把环境差异写死在镜像内

## 5. 部署前置条件

### 5.1 本地机器要求

- JDK 17
- Maven
- Docker / Docker Buildx
- `scp`
- `ssh`
- `curl`
- 如果目标机器用密码登录，还需要 `sshpass`
- 若要执行数据库覆盖导入，还需要 `mysqldump`

### 5.2 目标服务器要求

- 已安装 Docker 与 Docker Compose
- 具备可写部署目录
- 能够被当前执行部署的机器访问
- 端口已规划并放行
- 目标磁盘空间足够容纳镜像导入与容器重建

### 5.3 代码准备要求

- 当前分支代码必须可编译
- 部署前建议先执行最小回归测试
- 若包含数据库结构变更，必须先确认 SQL 升级方案

## 7. Stage 手工部署

### 7.1 准备环境文件

首次部署时先基于模板生成真实配置：

```bash
cp script/docker/env/stage.env.example script/docker/env/stage.env
```

随后补齐以下敏感项：

- 数据库密码
- 微信相关配置
- 远端登录方式
- 其他业务集成配置

### 7.2 执行部署

```bash
cd /Users/mac/项目/若伊部署
DEPLOY_ENV=stage bash script/docker/deploy.sh
```

或使用 `107` 兼容入口：

```bash
cd /Users/mac/项目/若伊部署
bash script/docker/deploy-to-107.sh
```

### 7.3 部署脚本的真实行为

`deploy.sh` 实际会做以下事情：

1. 校验环境变量文件存在且关键字段已配置
2. 执行 `mvn -pl yudao-server -am -DskipTests package`
3. 构建 `ruoyi-server:local`
4. 构建 `ruoyi-admin:local`
5. 导出镜像 tar
6. 上传镜像与环境文件到目标服务器
7. 远端执行 `remote-reload-app.sh`
8. 远端停止旧的 `server` / `admin` 容器
9. 远端加载新镜像并重建容器
10. 使用 `http://127.0.0.1:48080/actuator/health` 健康检查
11. 清理本地 tar 包、构建缓存，以及远端上传残留文件

### 7.4 部署完成后的检查项

建议至少检查以下内容：

- 前端首页可访问
- `actuator/health` 返回 `UP`
- 登录页可正常打开
- 关键菜单可正常进入
- 与外部系统集成的关键能力可调用

### 7.5 Stage 文件访问域名修复

`stage` 环境如果浏览器控制台出现头像、附件或图片资源跳到 `127.0.0.1:48080`，根因通常不是前端，而是数据库里的文件配置表 `infra_file_config.config.domain` 仍然写着本机回环地址。

修复方式：

1. 执行增量脚本 `sql/mysql/stage-file-url-fix.sql`
2. 重点确认 `infra_file_config` 中作为主配置或正在使用的配置，其 `domain` 已改成当前 `stage` 可访问地址，例如 `http://192.168.1.107:48080`
3. 历史 `system_users.avatar` 等已落库的完整 URL 也要一起替换，否则只改配置不会修复旧头像

验证 SQL：

```sql
SELECT id, master, storage, JSON_UNQUOTE(JSON_EXTRACT(config, '$.domain')) AS domain
FROM infra_file_config
ORDER BY id;

SELECT id, username, avatar
FROM system_users
WHERE avatar IS NOT NULL AND avatar <> '';
```

## 8. 生产环境部署

### 8.1 生产环境文件准备

```bash
cp script/docker/env/prod.env.example script/docker/env/prod.env
```

补齐生产机密后执行：

```bash
cd /Users/mac/项目/若伊部署
DEPLOY_ENV=prod bash script/docker/deploy.sh
```

### 8.2 生产部署建议

- 不要在生产环境默认使用密码登录，优先使用 SSH 私钥
- 部署前先在 `stage` 验证完全一致的代码版本
- 生产环境变更应带版本号、变更单或发布记录
- 若包含数据库变更，应先备份数据库再上线
- 生产环境建议固定时间窗口上线，并保留回滚方案

## 9. 数据库初始化与升级

### 9.1 首次初始化

首次拉起 MySQL 容器时，Compose 会挂载：

- `sql/mysql/ruoyi-vue-pro.sql`

该文件用于初始化主业务数据库结构与基础数据。

### 9.2 覆盖导入数据库

如需将本地数据库整体导出并覆盖到目标环境：

```bash
cd /Users/mac/项目/若伊部署
IMPORT_DATABASE=true DEPLOY_ENV=stage bash script/docker/deploy.sh
```

执行逻辑：

1. 本地 `mysqldump`
2. 远端启动 `mysql` / `redis`
3. 等待 MySQL 就绪
4. 远端删除旧库并重建
5. 导入新的 SQL Dump
6. 再启动 `server` / `admin`

### 9.3 增量 SQL 管理原则

增量 SQL 位于：

- `sql/mysql/*.sql`

典型脚本示例：

- `sql/mysql/kod-sso.sql`
- `sql/mysql/kod-dept-sync.sql`
- `sql/mysql/party-file-kod-upgrade.sql`
- `sql/mysql/system-meeting-room-init.sql`
- `sql/mysql/system-personal-schedule-init.sql`

建议执行顺序：

1. 先确认当前目标库版本
2. 按模块执行新增脚本
3. 执行后立即校验表结构和关键数据
4. 再部署应用代码

### 9.4 变更控制建议

- 不要把“结构升级”和“业务验证”混在一条不可回滚命令里
- 每个增量脚本都应可单独审阅
- 涉及可道云、党务文件、组织树等功能时，升级后必须做接口级验证

## 10. 可道云相关部署要点

### 10.1 SSO 配置入口

后端通过以下变量读取可道云配置：

- `KOD_SSO_ENABLED`
- `KOD_SSO_BASE_URL`
- `KOD_SSO_APP_NAME`
- `KOD_SSO_REDIRECT_URI`
- `KOD_SSO_CALLBACK_BASE_URL`
- `KOD_SSO_TENANT_ID`
- `KOD_SSO_AUTO_CREATE_USER`

### 10.2 地址配置原则

- `REDIRECT_URI` 必须指向用户真实可访问的前端域名或 IP
- `CALLBACK_BASE_URL` 必须指向用户真实可访问的后端域名或 IP
- 不要把 `127.0.0.1`、`localhost` 写进面向浏览器的公开地址

### 10.3 党务文件与可道云目录联动

若启用了党务文件可道云目录来源能力，部署后还需要确认：

- `party_file_kod_source` 相关字段已升级
- 目录来源记录已配置正确
- 服务账号模式可正常换取 token
- 可道云目录树可读取
- 可道云文件列表可读取
- 附件上传 / 预览 / 下载链路可用

## 11. GitHub Actions 自动部署

### 11.1 当前工作流

仓库内已有：

- `.github/workflows/deploy-stage.yml`

触发条件：

- `push main`
- 手动 `workflow_dispatch`

### 11.2 运行前提

该流程要求：

- 使用能够访问目标内网的 `self-hosted runner`
- runner 带标签 `ruoyi-deploy`
- 机器上已安装 Docker、Maven、JDK、SSH 能力

### 11.3 当前 Secrets 设计

工作流通过模板文件生成临时环境变量文件，并通过 GitHub Secrets 注入敏感值。

至少需要以下 Secrets：

- `DEPLOY_SSH_PRIVATE_KEY`
- `STAGE_MYSQL_ROOT_PASSWORD`
- `STAGE_WX_MP_APP_ID`
- `STAGE_WX_MP_SECRET`
- `STAGE_WX_MINIAPP_APP_ID`
- `STAGE_WX_MINIAPP_SECRET`

### 11.4 自动部署执行链路

1. checkout 代码
2. 安装 JDK
3. 由 `stage.env.example` 生成临时 `.stage.ci.env`
4. 注入 Secrets
5. 配置 SSH 私钥
6. 调用 `script/docker/deploy.sh`
7. 清理临时文件

## 12. 回滚方案

### 12.1 应用回滚

当前部署方式默认不会保留“带版本号标签”的镜像，因此严格意义上的一键镜像回滚能力较弱。

建议生产环境补充以下约束：

- 构建镜像时增加 Git SHA 或版本号 tag
- 远端保留最近 1 到 2 个稳定版本
- 发布记录中明确代码版本与镜像版本的对应关系

在当前体系下，可行的回滚方式是：

1. 切回上一个稳定 commit
2. 重新执行 `deploy.sh`
3. 用旧代码重新构建并覆盖发布

### 12.2 数据库回滚

若使用了 `IMPORT_DATABASE=true` 或执行了结构变更：

- 上线前必须先备份数据库
- 增量 SQL 应单独保留
- 需要有回滚脚本或人工回退方案

## 13. 常见故障排查

### 13.1 前端能打开，但接口全报错

优先检查：

- `ruoyi-server` 是否正常启动
- `actuator/health` 是否为 `UP`
- 后端端口映射是否正确
- 前端构建环境变量中的 API 地址是否正确

### 13.2 前端动态 chunk 加载失败

典型现象：

- 浏览器报 `Failed to fetch dynamically imported module`
- 静态资源返回 `text/html`

优先检查：

- Nginx 是否把旧 chunk 错误回退到 `index.html`
- `script/docker/nginx.conf` 是否为当前版本
- 浏览器缓存是否仍命中旧资源

### 13.3 可道云请求提示未登录

优先检查：

- `KOD_SSO_BASE_URL` 是否正确
- 服务账号模式是否启用
- 目录来源表中的 token 是否过期
- 回调地址是否与实际访问域名一致

### 13.4 浏览器出现 `127.0.0.1` 资源请求

这通常说明某条数据库中的 URL、头像地址或回调地址仍写成了本机回环地址。

处理方式：

- 查对应业务表中的 URL 字段
- 查系统用户头像地址
- 查前端或后端环境变量中的公开访问地址

### 13.5 上传文件到可道云目录失败

优先检查：

- `party_file_kod_source` 表字段是否完整
- 可道云目录来源记录是否启用
- `infra_file` 与 `party_file_kod_attachment` 是否成功落库
- 最近一次 `ruoyi-server` 日志中的 SQL 或外部接口错误

## 14. 发布检查清单

建议每次发布至少完成以下检查：

- 代码已合入目标分支
- 环境变量文件已核对
- 数据库升级脚本已评审
- 本地 Maven 构建成功
- 本地 Docker 构建成功
- 目标服务器磁盘与端口状态正常
- 发布后健康检查通过
- 登录、关键菜单、核心业务链路已抽样验证
- 如涉及第三方集成，已做真实接口验证
