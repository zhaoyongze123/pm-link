# 多租户与数据权限深度讲解

这份文档讲两个经常被混淆的能力：多租户和数据权限。

## 一、先讲结论

- **多租户**：你属于哪个租户。
- **数据权限**：你在这个租户里能看哪些数据。

它们不是一回事。

## 二、多租户是怎么进系统的

入口在 `YudaoTenantAutoConfiguration`：

`yudao-framework/yudao-spring-boot-starter-biz-tenant/src/main/java/cn/iocoder/yudao/framework/tenant/config/YudaoTenantAutoConfiguration.java`

它做了几件关键事情：

1. 注册租户上下文过滤器。
2. 注册租户安全过滤器。
3. 注册 MyBatis 租户拦截器。
4. 注册 Redis、MQ、Job 的租户传播能力。

### 1. 请求进来先带租户上下文

租户上下文不是业务表字段，而是请求上下文。

这样后续 SQL、缓存、消息都能知道当前租户是谁。

### 2. SQL 自动拼租户条件

`TenantLineInnerInterceptor` 会把租户条件自动加到 SQL 里。

这意味着：

- 业务代码不用每次手写 tenantId 条件
- 但也必须小心处理忽略租户的场景

### 3. 为什么有 `TenantIgnore`

像文件下载、公共回调、部分认证接口，不能硬套租户条件。

所以框架提供了 `@TenantIgnore`，让特定接口跳过租户过滤。

## 三、数据权限是怎么工作的

数据权限的核心注解是 `@DataPermission`。

它的意义是：

- 默认开启数据权限
- 可以显式关闭
- 可以指定启用或排除某些规则

这套机制通常用于：

- 按部门过滤数据
- 按角色过滤数据
- 按用户过滤数据

## 四、为什么登录权限查询要关闭数据权限

在 `AuthController.getPermissionInfo()` 上有 `@DataPermission(enable = false)`。

原因很直接：

- 查询登录用户本身、角色、菜单时，如果还套数据权限，可能把自己过滤掉
- 权限系统不能被自己关死

这是这套仓库里很典型的“权限系统的自举问题”。

## 五、你要掌握的 Java / 架构知识

### 1. AOP

`TenantIgnore` 和 `DataPermission` 都依赖横切能力。

### 2. 拦截器

MyBatis 拦截器会改 SQL，Web 过滤器会改请求上下文。

### 3. ThreadLocal

租户信息和登录用户信息大多通过上下文保存。

### 4. 横切关注点

租户、权限、日志、异常、缓存，都是横切能力，不应该散落在业务代码里。

## 六、你以后排查问题的顺序

1. 先看请求有没有拿到 tenantId。
2. 再看 SQL 有没有被租户拦截器改写。
3. 再看相关接口是不是误加了 `@TenantIgnore`。
4. 最后再看数据权限规则是不是把数据过滤掉了。
