## 建议落库模型

### 部门层级

- 顶层部门：`院部`
- 二级负责人层不单独建部门，作为审批链或岗位/角色使用：
  - `钱爱梅`
  - `邵燕萍`
  - `马倩`
  - `张华`
- 业务部门建议建为 `system_dept`：
  - `城市更新规划所`
  - `城市设计所`
  - `总体规划和专项规划所`
  - `乡村规划所`
  - `综合科（人事、行政、财务）`
  - `规划研究室`
  - `数字城市所`
  - `交通市政规划所`
  - `技术审查室`
  - `综合科（经营）`

### 岗位建议

- `院长`
- `副院长`
- `总工/师`
- `所长`
- `负责人`
- `普通员工`

### 角色建议

优先复用已有角色，不先扩角色：

- `super_admin`
- `dept_admin`
- `common_user`

只有在 BPM 确实要按“角色”指派审批人，而不能通过“部门负责人/多级负责人/发起人自选”实现时，再补业务角色。

## BPM 映射建议

### 通用节点

- `发起人`：流程发起人
- `部门领导（所长）`：使用“发起人部门负责人”
- `分管领导`：优先使用“发起人连续多级部门负责人”
- `院长`：继续使用“发起人连续多级部门负责人”，或对特殊节点单独指定角色/用户

### 例外节点

以下内容无法只靠组织树自动推导，后续需要单独配置：

- `张华、张龄、朱新捷直接到院长`
- `严己白` 从 `张华 / 张龄 / 朱新捷` 三选一
- `赖志勇到朱新捷`
- `李豪杰到何志华到张华`

## 已识别出的关键问题

### 1. 部门名称存在不一致

- 图片里是 `城市更新规划所`
- CSV 里是 `城市更新设计所`

后续生成 SQL 前必须统一口径，否则会导致部门树和导入账号不一致。

### 2. 图片人员与 CSV 人员不完全重合

图片里出现但 CSV 未直接匹配到的关键人员：

- `钱爱梅`
- `邵燕萍`
- `张华`
- `张龄`
- `苏莉`
- `罗翔`
- `何志华`
- `李豪杰`

已确认映射：

- `严己白` 对应 CSV 账号 `严己`
- `邵燕萍` 对应 CSV 账号 `邴燕萍`

### 3. 组织协作部门不宜直接入主部门树

CSV 里大量出现以下协作空间/虚拟分组：

- `公共空间`
- `科研小组`
- `案例库`
- `专家库`
- `博士后`

这些更像可道云协作目录，不建议直接映射成若依审批主部门，除非你明确要一起导入。

## 当前数据库现场

已核实 Docker 容器 `yudao-mysql` 中 `ruoyi-vue-pro` 库的 `tenant_id = 1` 当前不是样例树，而是一套旧组织：

- `总公司`
- `销售部`
- `项目部`
- `财务部`
- `行政人事部`

且目前仅有 `kodadmin`、`koddepartment` 等少量账号实际绑定了部门。

因此，初始化 SQL 应采用：

- 先清空 `tenant_id = 1` 旧部门树
- 再重建院内 11 个真实部门
- 再按已知账号回填用户部门、岗位、角色
